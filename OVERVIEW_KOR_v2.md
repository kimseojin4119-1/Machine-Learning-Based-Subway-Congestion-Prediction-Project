- 두 가지 ML 모델 탐색, XGBoost + RF 앙상블 모델 최종 선택

## 모델 비교
| Model | 검증 방식 | MAE | 총평 |
|---|---|---|---|
| XGBoost+RF 앙상블 | walk-forward 6개 시점 | 17.96 | 오염을 제거하니 실제 오차는 약 2배 |

### 전처리 단계
1. **승차 데이터만 필터링**: 원본은 승차/하차가 함께 있어 `승차` 구분만 남김
2. **주말 데이터만 추출**: 2024-12-01이 일요일인 점을 감안해, 요일(`weekday()`)을 계산해 토/일요일만 남기고 평일은 제거
3. **공휴일 제거**: 연도별 실제 공휴일(설·추석 연휴, 대체공휴일 등)을 날짜로 지정해 제외 — 주말이어도 공휴일이면 승차량이 비정상적으로 튀는 경향이 있어 이상치로 간주해 삭제
4. **새벽 시간대(1~4시) 값 0 처리**: 2021~2023년 데이터는 이 시간대 운행이 없어 0으로 강제 대입, 2024년 데이터는 원본 값을 그대로 사용
5. **날짜·역 기준 그룹화 합산**: 동일 (날짜, 역) 조합이 여러 행으로 나뉘어 있을 경우 24개 시간대 값을 합산해 하나의 행으로 정리
6. **역 단위 병합**: 위 10종의 외부 데이터를 `STATION_NO`(또는 연도+`STATION_NO`) 기준으로 순차적으로 left join

## 🤖 모델링

### 피처 구성
정적 피처(역/인구/POI/날씨) 14개 + 캘린더 피처(`is_sunday`, `Year`, `month`, `day`, `is_covid_period`, `season`, `is_special_event`) 7개 = 기본 21개. 재작업 과정에서 역×요일타입별 과거 8회 이동평균(`recent_avg_CNT01`~`recent_avg_CNT00`) 24개를 추가해 최종 **45개 피처**를 사용합니다. `is_special_event`는 EDA에서 확인한 대전 0시 축제·K리그 홈경기 날짜를 (역, 날짜) 단위로 표시한 이진 피처입니다.

### 모델 구조
- **XGBoost + RandomForest Voting 앙상블**: 24개 시간대(`CNT01`~`CNT23`, `CNT00`) 각각을 별도 모델로 학습, `VotingRegressor(weights=[2,1])`로 XGBoost에 더 높은 가중치 부여
- 하이퍼파라미터는 `GridSearchCV`(cv=3)로 탐색: XGBoost는 `n_estimators∈{200,400}`, `learning_rate∈{0.05,0.1}`, `max_depth∈{3,5}` / RandomForest는 `n_estimators∈{200,300}`, `max_depth∈{10,15,None}`, `min_samples_leaf∈{1,2}`
- 학습 시 샘플 가중치: 일요일 ×2.0, 코로나 기간(~2022.08) ×2.0, 출퇴근 시간대(7-9시, 17-19시) ×1.2 (곱연산으로 중첩)
- 이상치가 학습을 왜곡하지 않도록, 학습 타겟만 **역별 95백분위수로 캡핑**(윈저라이징) — 입력 피처나 실제 평가값은 그대로 두고 학습 타겟에만 적용
- **SARIMA(1,1,1)**, **LSTM** 두 가지를 비교 모델로 함께 구성

### 검증 방식: walk-forward
원본은 2024-11-24 단 하루로만 검증했지만, 재작업에서는 **최근 일요일 6개(2024-10-20, 10-27, 11-03, 11-10, 11-17, 11-24)** 를 순차적 테스트 시점으로 사용합니다. 각 시점마다 그 날짜 **이전** 데이터로만 학습하고, 하이퍼파라미터는 가장 이른 테스트 시점(10-20) 이전 데이터로 1회만 튜닝해 모든 폴드에 재사용합니다(폴드마다 재튜닝하는 비용을 줄이고 동일 조건에서 비교하기 위함).

### recency(이동평균) 피처의 효과 — 구암역 사례
역×요일타입(토/일)별로 `shift(1).rolling(8, min_periods=3)`로 과거 8회 평균을 계산합니다(`shift(1)`로 자기 자신은 제외해 미래 정보 누수가 없습니다). 예를 들어 구암역 12시(`CNT12`)는 직전 2번의 일요일(11/24=293명, 9/29=325명)이 우연히 높았는데, 이 피처가 있으면 모델이 "최근 8주 평균 141.75명"이라는 안정적인 신호를 함께 참고하게 됩니다. 이 피처 하나로 walk-forward hourly MAE가 17.96→**15.22**로, daily_total MAE가 299.1→**194.4**로 크게 개선되었습니다.

### LSTM 재구현
원본 LSTM은 timestep=1로 구성되어 있어 recurrence가 사실상 무의미했습니다. 재작업에서는 역별 **과거 6주(SEQ_LEN=6) 일요일 24시간 벡터**를 입력으로 다음 일요일의 24시간 벡터를 예측하는 진짜 시퀀스 모델을 구성했습니다. 22개 역을 하나의 모델로 함께 학습하며 역 정보는 4차원 임베딩으로 추가합니다(LSTM hidden size 64, Dropout 0.2, Adam optimizer lr=1e-3·weight_decay=1e-4, 150 epoch). TensorFlow가 재작업 환경(Apple Silicon, Python 3.13)에서 임포트만 해도 세그폴트가 발생해 **PyTorch로 대체 구현**했습니다.

### 재검증 과정에서 발견하고 고친 버그
| 문제 | 원본 | 수정 |
|---|---|---|
| train/test 오염 | 날짜로 분리한 학습 데이터를 곧바로 필터링 안 된 전체 데이터로 덮어써서, 검증 날짜가 학습에도 포함됨 | 매 검증 시점 이전 데이터로만 학습(walk-forward) |
| 단일 시점 검증 | 2024-11-24 하루만 검증 | 최근 일요일 6개로 walk-forward 검증 |
| 샘플 가중치 미반영 | 개별 모델에만 가중치를 주고 최종 voting 모델은 가중치 없이 재학습 → 무효화 | 최종 모델에 가중치가 실제로 반영되게 수정 |
| Feature Selection 임계값 불일치 | 출력은 `>0.01` 기준, 실제 선택은 `>0.005` 기준 사용 | 재작업에서는 피처 수 자체를 정리해 별도 selection 생략 |
| recency overfitting | 최근 1~2주 튐(예: 특정 역의 우연한 급증)에 모델이 과도하게 반응 | 역×요일타입별 과거 8회 이동평균(lag 피처, 미래 정보 누수 없음) 추가 |
| LSTM이 사실상 시퀀스를 안 씀 | timestep=1로 구성되어 recurrence가 무의미 | 과거 6주 시퀀스 + 역 임베딩 기반으로 재구현 (TensorFlow가 재작업 환경에서 크래시해 PyTorch 사용) |

## ⚖️ 모델 비교 및 선정

위 [모델 비교](#모델-비교) 표를 동일한 walk-forward 6개 시점 기준으로 다시 풀어보면:

| 모델 | hourly MAE | daily_total MAE |
|---|---|---|
| 앙상블 (원본, train/test 오염 버그) | 9.51 (신뢰 불가) | - |
| 앙상블 (버그 수정) | 17.96 | 299.1 |
| **앙상블 (버그 수정 + recency) ✅** | **15.22** | **194.4** |
| SARIMA(1,1,1), 일 총승차량만 | - | 190.5 |
| LSTM (PyTorch, 진짜 시퀀스) | 18.30 | 226.3 |

버그 수정과 recency 피처 적용 후 **XGBoost+RF 앙상블(hourly MAE 15.22)** 을 최종 모델로 유지했습니다.

- **SARIMA와의 비교**: 일 총승차량 기준으로는 190.5 vs 194.4로 거의 대등합니다. 이는 앙상블이 24개 시간대를 각각 예측한 뒤 합산하는 방식(bottom-up)이라 시간대별 오차가 누적되는 반면, SARIMA는 총량을 직접(top-down) 예측해 그런 누적이 없기 때문으로 해석됩니다. 다만 SARIMA는 시간대별 세부 예측을 제공하지 못해, "시간대별 예측"이 요구사항인 이 프로젝트에는 맞지 않았습니다.
- **LSTM과의 비교**: 처음부터 다시 구현한 진짜 시퀀스 LSTM(hourly MAE 18.30)은 이 정도 데이터 규모(역당 약 200개 관측)에서는 recency 피처를 더한 트리 앙상블보다 나은 성능을 보이지 못했습니다. 딥러닝이 항상 우월한 것은 아니며, 데이터 규모에 맞는 모델 선택이 중요하다는 것을 확인했습니다.

## 📊 결론

1. **역별 우선순위를 둔 배차 조정이 필요합니다.** 역별 규모 차이가 최대 15.7배(대전역 vs 갑천역)로 시간대·날씨보다
   훨씬 강한 변수이므로, "시간대별 전체 증차"보다 상위 몇 개 역(대전·유성온천·중앙로·반석·시청)에 자원을
   집중하는 편이 효율적일 수 있습니다.
2. **행사 캘린더와 연동한 임시 증차 체계가 필요합니다.** 대전역·중앙로역(대전 0시 축제)과 월드컵경기장역
   (K리그 홈경기)처럼 특정 일정에 따라 승차량이 6~10배까지 급변하는 역은 정기 배차만으로는 대응이 어렵습니다.
   실제로 이런 이벤트 날짜를 피처로 반영(`is_special_event`)하자 해당 역의 예측 오차가 크게 줄었습니다
   (월드컵경기장역 12/1 예측: 780명(오류) → 112명(수정) → 실제 오차 9.2로 최저 수준).
3. **예측 오차는 낮~저녁 시간대(11~17시)에 집중됩니다.** 새벽·심야는 오차가 거의 없는(MAE 0~5) 반면, 이 시간대의
   배차는 예측치보다 다소 여유 있게 운영하는 것이 안전할 수 있습니다.
4. **모델은 "과거 패턴 재현"에는 강하지만 "학습 범위 밖 미래로의 외삽"에는 한계가 있습니다.** 구암역 사례처럼,
   실제 운영에서는 예측치를 그대로 신뢰하기보다 최근 몇 주 흐름과 함께 교차 확인하는 절차를 두는 것이 안전합니다.

## 📝 배운점

- **검증 지표를 그대로 믿지 않는 습관**: 제출 당시 보고했던 Test MAE 9.51은 train/test 분리 코드의 버그로 인한
  낙관적인 값이었습니다. 같은 노트북이 함께 출력했던 Cross-Validated MAE(19.79, 오염 없는 값)가 사실 더 정확한
  신호였는데, 당시에는 이를 놓쳤습니다. 이후 walk-forward 검증과 실제 미래 데이터 비교로 이중 확인하는 습관의
  중요성을 배웠습니다.
- **이상치는 지우기 전에 원인을 찾아본다**: 데이터에서 발견한 극단치를 단순히 노이즈로 처리하지 않고, 날짜·역을
  특정해 실제로 검색해보니 대전 0시 축제·K리그 홈경기와 정확히 일치했습니다. 이후 이 정보를 피처로 활용해
  모델 성능을 개선할 수 있었습니다.
- **모델별로 잘하는 지표가 다를 수 있다**: "앙상블이 SARIMA/LSTM보다 항상 낫다"는 처음의 결론은, 총 승차량
  기준으로 보면 SARIMA와 거의 대등하다는 걸 재검증 후에 알게 됐습니다. 평가 지표(시간대별 vs 총량)에 따라
  결론이 달라질 수 있다는 것을 배웠습니다.
- **feature engineering으로 못 잡는 문제도 있다**: recency(최근 이동평균) 피처는 과거 시점 재현에는 큰 도움이
  됐지만, 학습 데이터 범위 밖(2024년 12월)으로의 진짜 외삽에서는 한계가 있었습니다. 트리 기반 모델의 근본적인
  한계를 실측 데이터로 확인한 경험이었습니다.
